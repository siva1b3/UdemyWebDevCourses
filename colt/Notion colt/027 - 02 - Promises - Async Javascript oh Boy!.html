283. Creating Our Own Promises

A promise can be in one of three states:

  • Pending: Initial state, neither fulfilled nor rejected.
  • Fulfilled: The asynchronous operation completed successfully.
  • Rejected: The asynchronous operation failed or encountered an error.

n a Promise, the state is initially undetermined or pending. The state can transition to either fulfilled (resolved) or rejected, and this transition is determined by whichever function among resolve and reject is called first inside the Promise's callback function.

  • If you call resolve() first inside the callback, the Promise transitions to the fulfilled state, indicating successful completion of the asynchronous operation.
  • If you call reject() first inside the callback, the Promise transitions to the rejected state, indicating an error or failure in the asynchronous operation.

Errors in function that is returning the promise:-

if an function that returning a promise is having some error then — promises will stay in pending status permanently as the asynchronous function was causing the error instead of calling either “reject (or) resolve” functions

When you call resolve and encounter an error inside the resolve function, the behavior depends on how you handle that error within the resolve function.

Here's what typically happens:

  1. Error within the resolve function:

    If an error occurs inside the resolve function, it will throw an error just like any regular JavaScript code would. For instance:

    const promise = new Promise((resolve, reject) => {
    try {
        // Some operation that throws an error
        throw new Error('An error occurred inside resolve.');
        resolve('Success!');  // This line will not be executed
    } catch (error) {
        reject(error);  // Reject the promise with the error
    }
});

    In this case, the error will be caught within the catch block, and the promise will be rejected with the error that occurred inside the resolve function.

  1. Handling errors within the resolve function:

    You can also handle the error inside the resolve function and still resolve the promise with a different value. For example:

    const promise = new Promise((resolve, reject) => {
    try {
        // Some operation that might throw an error
        const result = performSomeOperation();
        resolve(result);  // Resolve the promise with the result
    } catch (error) {
        // Handle the error and still resolve with a different value
        resolve('Fallback value due to an error.');
    }
});

    In this case, even if an error occurs, you're handling it within the resolve function and still resolving the promise with a fallback value.

So, calling resolve and encountering an error inside the resolve function won't prevent the error from propagating; it will throw an error that you can handle within a try...catch block or choose to resolve the promise with a fallback value.
